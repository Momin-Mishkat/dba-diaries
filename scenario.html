The 2 A.M. Restore: An RMAN Recovery Scenario I Don't Want to Repeat

A corrupted datafile, a partially-down application, and a decision I had about four minutes to make. Here's exactly how the recovery went, command by command, and what I'd do differently.

This one didn't start with a ticket. It started with my phone lighting up at 1:52 a.m. because the monitoring alert had already beaten the app team to it. I want to walk through this one in detail — not the sanitized "and then we restored from backup" version, but the actual sequence, including the fifteen seconds I spent staring at the screen deciding what NOT to do first.

1. The page — ORA-01578 in the alert log

The alert log had one line that mattered, sitting quietly above a stack of session errors that were really just symptoms. This is the line that tells you it's not a slow query problem — it's a block problem.

ORA-01578: ORACLE data block corrupted (file # 7, block # 1045) ORA-01110: data file 7: '/u02/oradata/PRODDB/users01.dbf'

First rule I've learned the hard way: don't touch anything yet. Don't restart the instance, don't bounce the listener, don't let anyone "just try running it again." Corruption doesn't fix itself, and a bad restart can turn a contained problem into a much bigger one.

2. Confirm the blast radius before doing anything else

One corrupt block reported in the alert log doesn't tell you if it's one block or fifty. I ran a validate against the whole datafile before deciding on an approach, because the recovery method genuinely depends on the answer.

RMAN> VALIDATE DATAFILE 7; SQL> SELECT file#, block#, blocks, corruption_type FROM v$database_block_corruption;

In this case it came back with three contiguous blocks, all in the same datafile, all in the USERS tablespace. Not the whole datafile — just a small, localized patch. That single fact changed everything about the next step.

3. Block-level recovery vs. full datafile restore

This is the decision point people rush past. A full datafile restore is safer in the sense that it's simpler to reason about, but it also means restoring and rolling forward the entire file — more I/O, more downtime, more time the tablespace sits offline. With three isolated blocks and a recent backup available, block-level recovery was the faster and lower-impact option, and Oracle supports exactly this case.

4. Taking only what's necessary offline

Because the corruption was confined to one datafile in a tablespace with other datafiles, I didn't have to take the whole application down — just accept that a slice of USERS data was unavailable for the few minutes the recovery took. That's the difference between a partial degradation and a full outage, and it's worth checking before you assume you need a maintenance window.

SQL> SELECT file_id, tablespace_name FROM dba_data_files WHERE tablespace_name = 'USERS';

5. Running the block recovery

With the exact file and block numbers in hand from the validate step, BLOCKRECOVER goes straight to the source — it restores and applies redo to just those blocks instead of the entire file.

RMAN> BLOCKRECOVER DATAFILE 7 BLOCK 1045, 1046, 1047;

If the corruption had come back covering the whole datafile instead of a tight cluster, this is where I'd have switched plans — taken the datafile offline, restored and recovered the full file, then brought it back online. Worth knowing both paths before you're standing in front of the decision at 2 a.m.

RMAN> SQL 'ALTER DATABASE DATAFILE 7 OFFLINE'; RMAN> RESTORE DATAFILE 7; RMAN> RECOVER DATAFILE 7; RMAN> SQL 'ALTER DATABASE DATAFILE 7 ONLINE';

6. Validate again before declaring victory

A recovery that "completed successfully" gets exactly the same skepticism I give a backup that "completed successfully." I re-ran the validate and checked the corruption view again before telling anyone the incident was closed.

RMAN> VALIDATE DATAFILE 7; SQL> SELECT count(*) FROM v$database_block_corruption;

7. The root cause — and why "disk issue" isn't a real answer

The storage team's first answer was "a disk issue," which tells you nothing. It took a joint look at the storage controller logs to find the actual trigger — a firmware-level write error during a maintenance window earlier that night that never got flagged up to us. Worth pushing for the specific answer, because it directly decides whether this happens again next month.

8. What I changed after this one

Three things went in immediately: block change tracking, so future incremental recoveries are faster; a proactive weekly VALIDATE DATABASE job instead of waiting for corruption to announce itself in the alert log; and a direct alert wired to page on any ORA-01578, ORA-01578-adjacent, or v$database_block_corruption row appearing — instead of relying on someone reading the alert log at the right moment.

RMAN> CONFIGURE BLOCK CHANGE TRACKING ON;
···
The backup isn't the safety net. Knowing exactly which recovery command to run, for which scenario, before you're standing in the middle of one — that's the actual safety net.

The whole incident, from page to all-clear, took about thirty-five minutes. Most of that was the validate steps, and I don't regret a single minute of it — a recovery you haven't verified is just a guess with extra steps. What took years to build wasn't the RMAN syntax; it was the judgment to know which of the two recovery paths a given corruption actually calls for.
